# TUI interaction depth

All 6 refined 2026-07-12 by 4-assessor fan + judge; ranked value-for-cost: yank > board peek > jumplist > undo > visual-mode bulk > palette.

## Visual-mode multi-select for bulk operations
Status: accepted (M6 stretch)
Why: Grooming a backlog one item at a time is the #1 tedium JIRA users flee; lazygit's range-staging shows how natural bulk ops feel in a panel TUI.
Sketch: `v` toggles visual mode — v1: Tree only, flat homogeneous selections; `j`/`k` extends, `Space` toggles per item (palette gets `x`). State-agnostic actions ship first (`a`, labels); bulk `m` offers only the intersection of reachable states across the selection (empty → popup explains why, per-item validity shown). Confirmation popup pre-validates all N and lists skips; execution is explicitly non-atomic skip-and-continue over the same single-item `internal/core` calls (one semantic commit each), ending in a per-item outcome popup (moved / skipped: reason). Selection clears on `r`/reindex; hint bar swaps to visual-mode bindings from the same keymap table.
Caveats: N commits ⇒ N undos — no bulk rollback exists or should be promised; Board cross-column selection deferred (semantically murky).

## Undo last mutation (`u`) — semantic inverse via core
Status: accepted (M6 stretch)
Why: JIRA cannot cleanly undo a state change; kira can, because every auto-mode mutation is one structured commit. Instant, trustworthy undo is the git-native delight moment.
Sketch: `kira undo` in `internal/core` (shared CLI/TUI per the single-write-path rule): parse the target `kira:` commit's structural diff (machinery `kira log` already has), compute the semantic inverse, apply it through the same core call with an explicit undo flag → a normal validated `undo:` commit. Never raw `git revert` — that bypasses core, can land a state the transition adjacency map forbids, and 3-way-conflicts whenever a later commit touched the same item file. v1 gate: target commit is HEAD (or provably no later commit touched the file) and worktree clean; refuse with a one-line reason otherwise, incl. when the inverse transition isn't computable or the field has since changed. No "refuse if pushed" gate — this is a forward commit, always history-safe.
Caveats: unavailable in `commit.mode: manual` (mutations ride inside user commits — say so in the popup); "revert applies cleanly" and "result is a valid kira state" are independent properties — never conflate them; redo = undo of the undo, same gate.

## Yank keys: `y` ID, `Y` structured forms
Status: implemented (M4 WP-4.6)
Why: The most common exits from a ticket TUI are "paste the ID into a commit message / branch name / PR description" — lazygit's copy-SHA muscle memory.
Sketch: `y` copies `KIRA-142`; `Y` opens a tiny picker: git trailer (`Kira-Ticket: KIRA-142`), branch slug (`kira-142-fix-race-in-snapshot`), markdown link — forms from one formatting helper shared with `kira show --format` so CLI and TUI never drift. Clipboard chain is environment-aware, not bare LookPath: always emit OSC 52 (tmux ≥3.3 passthrough-wrapped when `$TMUX`); external tool only when its environment supports it (`pbcopy` on darwin, `wl-copy` iff `$WAYLAND_DISPLAY`, `xclip`/`xsel` iff `$DISPLAY`). Hint-bar toast states the mechanism honestly: "sent OSC 52 (unverified)" vs "copied via xclip".
Caveats: OSC 52 has no ack channel and is off by default in tmux (`set-clipboard`) and some vte terminals — a confident "copied!" toast can lie; bare LookPath finds a remote `xclip` on a headless ssh box that then blocks/fails without `$DISPLAY`.

## Jumplist navigation: `Ctrl-o` / `Ctrl-]`
Status: implemented (fold into WP-4.1)
Why: Following parent links, linked commits, and board→detail hops loses your place; vim users' fingers already know the fix.
Sketch: The nav stack WP-4.1 already builds (view + focused item + scroll offset) becomes a bounded jumplist; `gp`, `Enter`, `1`/`2`/`3` push onto it. `Ctrl-o` walks back, `Ctrl-]` forward — not `Ctrl-i`, which is byte-identical to `Tab` (0x09) and §2 binds `Tab` to pane cycling; `Ctrl-i` becomes an alias only when kitty-keyboard-protocol disambiguation is negotiated. Staleness: after `r`/reindex, entries whose item vanished are dropped/skipped (degrade to nearest valid entry), never restored dangling. Pure model state — no core or storage change.
Caveats: entries hold ULIDs + scroll offsets that go stale across refresh; the `Ctrl-i`≡`Tab` encoding constraint is recorded in 05-tui §9.

## Action palette (`x`): fuzzy every-action menu
Status: accepted (M6 stretch)
Why: k9s/gh-dash discoverability gap: `?` shows keys, but a newcomer wants "type what you mean". A palette bridges casual users to muscle memory without a second command system.
Sketch: Extend keymap rows to `{key, description, action-msg}` (hint bar and `?` ignore the new field — the single source is strengthened, not forked); the palette (`x` — `Space` goes to visual mode) is a strict fuzzy-filtered projection of those rows, dispatching the identical msg the keypress would. Selecting "move" opens the normal move popup — zero inline transition expansion. Fuzzy-typeahead via bubbles textinput; each row shows its key, teaching the binding as it's used.
Caveats: modest marginal value over hint bar + `?` + `:` bar — lowest priority of the six; reverts to rejected the moment scope creeps toward per-transition rows or context predicates (that would be the second command system §2/§7 forbid).

## Board detail-peek toggle (`p`)
Status: implemented (fold into WP-4.3)
Why: gh-dash's split preview is why people call it "professional": triaging a board without bouncing into and out of a full overlay per card keeps flow.
Sketch: `p` on Board docks the existing detail renderer (§5) as a right-hand pane — a third mount of the same renderer — live-following card focus; toggle is session-scoped, `Enter` still opens the full overlay. Minimum-width guard: below threshold the peek auto-collapses (hint-bar note) rather than crushing columns below card-title width — Board columns are unbounded (`workflows.<type>.states`) and §9 targets 80–100-col nvim `:terminal` splits. Reconcile §2/§4: decide whether `Tab` focus-cycling extends to the docked pane (Board is currently pane-less).
Caveats: without the width gate, `p` yields a strictly worse board than the `Enter` overlay it complements; teatest needs a narrow-terminal snapshot case.

# Visual polish

All 6 refined 2026-07-12 by 4-assessor fan + judge; ranked value-for-cost: progress bars > WIP tinting > theme struct > staleness heat > responsive; density rejected. Theme struct is the dependency root (heat, WIP tint, bar fill all consume its slots) — sequenced first in M4; board-peek gate, medium-tier threshold, and progress-bar collapse share one min-width constant.

## Semantic theme system with adaptive light/dark
Status: accepted (first M4 TUI WP — WP-4.0; `ui.theme` product surface → M6 stretch)
Why: The difference between a screenshot people share and "default ANSI on black" is a coherent palette; hardcoded colors break on light terminals and look dated next to lazygit/gh-dash theming.
Sketch: M4 ships only the internal `Theme` struct — semantic slots (`category.todo/doing/done`, `priority.P0..`, `accent`, `dim`, `border`, `heat.warm/hot`), one default built on `lipgloss.AdaptiveColor`, enforced as the only permitted home for color literals from the first lipgloss call site. The entire user-facing surface (`ui.theme:` key, curated variants, inline slot-override map) is deferred to post-dogfood M6. `ui.background: auto|dark|light` escape hatch with explicit dark fallback when detection fails — AdaptiveColor's OSC-11/COLORFGBG probe is exactly the nested-terminal query §9 forbids assuming, so "adaptive" degrades to a guess in nvim `:terminal`, the primary habitat.
Caveats: teatest pins a fixed termenv color profile + background so goldens are environment-independent; slots share only a name taxonomy with 06 §4's nvim highlight groups — nvim colors defer to the user's colorscheme by design, so frontends will (and should) disagree on what DONE looks like.

## Epic progress bars in tree rows
Status: implemented (fold into M4 tree WP — WP-4.1)
Why: "3 of 7 done" at a glance is the most information-dense pixel-row an epic can carry — the screenshot moment; JIRA buries it two clicks deep.
Sketch: Epic rows in Tree render a fixed-width mini-bar + fraction after the title: `▰▰▰▱▱▱▱ 3/7`, from the recursive completion the telemetry engine (08) already computes per epic — one indexed query, no new data. Fixed display width in both glyph and ASCII (`[###----] 3/7`) forms per the §8 icon-table convention, so column alignment holds in either mode. Theme slot `category.done` colors the filled cells.
Caveats: the fraction adopts 08-telemetry's dropped-resolution convention exactly (dropped excluded from numerator, included in denominator) or tree bar and Stats visibly disagree on the same epic; recursive counts are fetched once per refresh (`r`) and cached in the model — never per `View()` frame, or a deep tree runs N index queries per keystroke; bar+fraction cost ~11 cells of title width in an 80-col nvim-split tree pane — collapse below the shared min-width constant (same one as WP-4.3's board-peek gate).

## Staleness and due-date heat
Status: accepted (M6 stretch, gated on the Theme struct; smallest promotion candidate post-dogfood)
Why: A board where every card looks equally urgent hides the two that are rotting; k9s-style graduated age color answers "what needs a look?" without reading a single date.
Sketch: Each row/card gets a dim right-aligned relative age (`3d`, `2w`) from `updated`, graduating dim→`heat.warm`→`heat.hot` (Theme struct slots) past config thresholds (`ui.stale_after: [7d, 21d]`; unset disables; duration strings validated by the config schema). `due` within 48h renders warm, overdue hot; categories `done` exempt. Pure render-time math over indexed columns (`updated`/`due`) — no per-item git-history reads, fast list path untouched. Time flows through a clock injected at the refresh tick (consistent with §7's no-watcher stance; frozen in teatest with pinned fixture dates); an ASCII/no-color affordance (e.g. `3w!` for hot) per the §8 dual-form convention keeps the signal visible in `--plain`/no-color contexts.
Caveats: without clock injection every age-bearing golden rots daily in CI; `updated` bumps on *any* write — a comment resets heat — so the signal measures last touch, not last progress; the honest signal (last state transition) needs derived-event data and is rightly out of scope.

## Density modes: compact and cozy
Status: rejected — the designed §3/§4 one-line rows *are* compact (cozy halves visible rows in exactly the 80×24 nvim-split target §9 names); "no new data" is false (comment count isn't an index column, so an index-schema change hides in a "two templates" sketch); worst snapshot multiplier of the six; the persistent detail pane + board peek already carry cozy's content for the focused item. Revisit only if dogfooding proves one-line rows starve users.
Why: A 20-ticket backlog wants metadata-rich rows; a 300-ticket one wants one line each — gh-dash earns "professional" largely from its tight, aligned two-line rows.
Sketch: `ui.density: compact|cozy` config default + a runtime toggle key. Compact: one line — glyph, ID, title, state. Cozy: a second dim line — owner · labels · due · comment count. One row model, two lipgloss templates, applied uniformly to Tree rows and Board cards; teatest snapshots per mode. Detail pane unaffected.

## Responsive layout breakpoints
Status: accepted (wide + medium tiers core M4, fold into WP-4.1; narrow tier → M6 stretch)
Why: Half this tool's life is inside an nvim `:terminal` split or tmux pane at 80×24 (§9 names nested terminals a tested target); panes that crush into garbage there read as amateur. Miscategorized as polish — §3's tree+detail split is two crushed 40-col panes at 80, so the medium tier is table stakes.
Sketch: On `WindowSizeMsg`, pick a layout tier. Core M4 in WP-4.1: wide (≥120 cols — tree+detail split, all board columns) and medium (Tree's detail pane hidden below the shared min-width constant — same one as WP-4.3's board-peek gate; `Enter` reuses Board's existing overlay mount of the same §5 renderer). M6 stretch: narrow (<80 — single paged board column full-width, `h`/`l` repages, header paginates `◂ IN_PROGRESS 2/4 ▸`) — a new interaction model, not a view switch. Tiers are *not* pure view-layer switches: they change `Tab` focus-cycling and `Enter` bindings, so per-tier keymap deltas live in the §7 single keymap table (hint bar and `?` stay single-sourced).
Caveats: golden budget capped at one snapshot per tier per view; icon-vs-ASCII variants at the wide tier only.

## WIP-limit pressure tinting
Status: implemented (fold into WP-4.3 — spec debt, not stretch: 02 §6 already promises board colouring of over-limit columns at M4)
Why: A WIP limit that only warns at `move` time (WP-1.5.4) is invisible the rest of the day; a column header sitting at `4/3` in amber nags exactly when you look at the board.
Sketch: Board column headers always render `n/wip` when `wip:` is configured (plain count when not) — no header-width jump at the breach boundary, pressure visible early: neutral below limit, `heat.warm` at count == wip, hot + ASCII `!` when over (the original `≥`-warm alarms a legal at-cap state). The numerator is the **global per-state count from the index, never the rendered subset** — `kira board <epic-id>` and an active `/` filter are first-class modes where count-what-you-render shows a calm 2/3 while the real column sits 5/3. Read-only render over config + index counts; pairs with WP-1.5.4's move-time warning so board and CLI tell one story.
Caveats: `wip:` is per-workflow-per-state — define whose limit applies when types with different `wip:` values share a column key, or `4/3` can be nonsense; heat colors come from the Theme struct's `heat.*` slots — sequence after WP-4.0 or this becomes the first hardcoded literal the theme discipline exists to forbid.

# Onboarding / first-run

All 6 refined 2026-07-12 by 4-assessor fan + judge; ranked value-for-cost: error contract > empty states > seed config > topic help > tour > guided init. Cross-cutting rulings: in `--json` mode structured errors (`{error, hint, code}`) go to stderr and stdout stays empty on failure — a stdout error object requires an explicit ratified 04-cli §7 amendment (binds the error contract and all future error-surface work); non-interactive bare `kira` in an uninit repo keeps exit 3 forever — only tty sessions get the teaching screen; redundancy cluster (seed config / topic help / guided init): seed config is the canonical config tutorial, guides don't restate it, wizard previews reuse its comment blocks — ship seed config first, judge guided init against observed friction; teatest goldens are test-owned and frozen — the tour seeder is CI-validated but never regenerates goldens; help altitude rule (`--help` = flags, `help <topic>` = concepts, TUI `?` = keys) prevents a second help system.

## Empty-state screens that teach
Status: implemented (M4 — fold Tree/Board/Stats states into the view WPs; uninit-repo screen split out as separate M4 item WP-4.7)
Why: Bare `kira` is the advertised entry point, and a brand-new project shows a blank tree — the single highest-leverage screen for retention is the one every user sees first.
Sketch: Tree/Board/Stats empty states ship as pure view-layer strings keyed off counts the model already holds, one teatest snapshot each: Tree with zero items shows "No tickets yet — `n` create · `:` command · `?` keys"; Board shows the configured workflow column names plus the same hint; Stats explains what will appear and when. The uninitialized-repo state (no `.kira/`) is split into its own WP — it forces a storeless TUI boot mode and a new interactive write entry point, and the expensive 10% must not block the free 90%. When it lands: tty sessions get a screen offering init via an explicit typed confirmation listing exact side effects — creates `.kira/`, makes a commit (`init` always commits, §3) — never bare `Enter`; it calls the same core init path.
Caveats: non-tty bare `kira` with no `.kira/` still exits 3 per the exit-code table — no CLI-contract leak; the tty-gated semantic change to bare `kira` must be documented.

## `kira init` guided mode
Status: accepted (M6 stretch, contingent: build seed config + topic help first, promote only if dogfooding shows first-run config friction; explicit `--guided` opt-in only)
Why: The default config encodes real decisions (workflow states, commit mode, priorities) that a newcomer can't evaluate from flag docs; a 60-second wizard converts them from "mystery YAML" to choices they made.
Sketch: Explicit opt-in — `kira init --guided`/`-i` — never a tty-sniffed default: pty-allocating CI, expect scripts, and muscle memory all hit a default wizard, and opt-in kills the bug class (making a `--defaults` escape moot). Trimmed to the two genuinely hard-to-reverse/evaluate decisions: workflow preset and commit mode (even the founder finds commit-mode non-obvious per the DESIGN open questions); project key stays a derived default, hooks install becomes a printed next-step hint. Each step's YAML preview reuses the seed config's commented blocks verbatim — one source of truth, no restated tutorial. Output is the same scaffold + one commit as today; the plain path stays byte-identical.
Caveats: bubbles is in-tree post-M4, making this a thin reuse; its value is 60–80% covered by seed config + topic help — that redundancy drives the demotion and the contingency.

## Self-documenting seed config
Status: implemented (M3.5 WP-3.5.2)
Why: Feature discovery for a config-driven tool happens inside config.yaml; a minimal seed file hides sprints, filters, WIP limits, and validators until the user rereads 20K words of docs.
Sketch: `init` writes the chosen config plus every unused optional block as commented-out, ready-to-uncomment examples with a one-line explanation and doc link (`# filters: {mine: "owner=me AND state!=DONE"} — saved queries for list --filter`). The original sketch guarded the wrong write path: `config edit` ($EDITOR on the raw file) is trivially safe, but `kira config set` (dotted-path programmatic write, §3) strips every comment under a naive yaml.v3 marshal — the tutorial self-destructs on the first `config set commit.mode manual`. Require comment-preserving yaml.v3 `Node`-level editing in `config set`. Two goldens: the sketched parse-equivalence golden (comments never carry semantics — inertness) plus a preservation golden asserting a `set` changes exactly the target line and keeps every comment block (survival) — both needed.
Caveats: Node-level surgery is fiddly (anchors/styles) — accepted as the feature's real cost; kira authors this file itself, so anchors and exotic styles are absent in practice; 03-storage §3 byte-stability covers tickets only — the preservation golden becomes the config-side invariant.

## Error contract: what · why · next action
Status: implemented (M3.5 WP-3.5.1)
Why: The first error message is the product's real first impression; "invalid state" vs "REVIEW is not reachable from DONE (allowed: REOPENED); use --force to override" is the gap between abandoned and adopted.
Sketch: Three-part human format on stderr ships now — what failed, why (with the offending value), one copy-pasteable next action; human output is explicitly unfrozen (§1). In `--json` mode the structured error `{error, hint, code}` goes to **stderr** and stdout stays empty on failure — failure stdout is contractually silent (scripts distinguish failure by empty stdout + exit code), so a stdout error object is permitted only as an explicit ratified §7 amendment with new goldens in the same change. One render helper in `internal/core`, every command's failure path routed through it; the exit-code table carries the error class. Typo suggestions (edit distance over item IDs/aliases, state names, label/people vocab, query field names, mirroring cobra's command suggestions) run only on the failure branch and only over vocabulary already in memory for the failed validation — never load the index solely to suggest. testscript goldens pin the top-10 first-run errors.
Caveats: permanent value, not onboarding-only — unanimous accept, highest rank of the six.

## Topic help: `kira help <topic>` rendered guides
Status: accepted (renderer note superseded: glamour was replaced by the in-tree goldmark plain-text renderer, which serves this directly; `query` + `sync` guides gate the public release, remaining guides at M7 WP-7.6)
Why: gh's `gh help environment`-style topic pages are what let a terminal tool ship without sending users to a browser; kira's query grammar and sync model both need prose, not flag lists.
Sketch: Embedded (go:embed) markdown guides rendered with glamour via `kira help <topic>`; `kira help` lists them under the cobra command list. v1 scopes to the two guides that earn their keep — `query` and `sync`; `workflow`/`config`/`hooks` follow. Guides are authored terminal-first (no images, glamour-safe links). Canonicality declared once: design doc = spec for implementers, guide = canonical user prose, web docs = generated from the guide — never three copies of the query grammar. Altitude rule recorded in 04-cli: cobra `--help` = flags, `kira help <topic>` = concepts, TUI `?` = keys.
Caveats: the real cost is authorship — docs/design/*.md are implementer prose, never embed them; "same files are the web docs" over-promises — embedded guides version with the released binary while web publishes from HEAD, so either build the site per-tag or accept release-lag drift.

## `kira tour`: disposable sandbox playground
Status: accepted (M6 stretch — launches the TUI; the internal seeder may land at M4 for demo GIFs)
Why: The only safe way to try a tool that commits to your repo is someone else's repo; a sandbox converts "will this pollute my history?" hesitation into a five-minute hands-on tour.
Sketch: Built inside-out: the seeder is an internal fixture package exercised by CI — a test runs it and validates the result via `kira doctor`/schema checks, so schema drift breaks the build — but it never serves as the teatest golden corpus; snapshot tests own their frozen fixtures. `kira tour` itself is a thin public wrapper promoted post-M4: temp git repo, init, seeded realistic backlog (one epic, ~6 tickets across states, a comment thread), launch the TUI. Seeds exclusively through public CLI commands; the "linked commit" goes through the real hook flow or is cut — never fabricate git history the product wouldn't produce; all sandbox git ops pass `-c user.name=… -c user.email=…` (the brand-new-machine target user often has no git identity configured); on quit print the sandbox path and an explicit `rm -rf <path>` cleanup line.
Caveats: the value is the seeded realistic backlog plus the "can't pollute my history" guarantee, not keystroke-saving; demo-edit → golden-churn coupling is exactly what the seeder/golden fixture split prevents.

# Publishing readiness

All 6 refined 2026-07-12 by 4-assessor fan + judge; ranked value-for-cost: release pipeline > compat policy > version handshake > README > vhs tapes; changelog rejected. Cross-cutting rulings: one install story — pipeline, README matrix, and the guard's install-hint text name only curl-binary + `go install` at v1; every new `--json` shape (incl. `version`) enters the frozen golden corpus in the change that ships it, and `json_contract` is asserted by that suite, never hand-maintained; the first tag is a one-way gate — `min_version` guard, codec write-refusal, and attestation must be aboard the first tagged binary, since nothing here is retrofittable to already-shipped binaries; dependency chains: pipeline ldflags → version handshake → guard's install hint, and tour seeder (M4) → vhs tapes → README hero GIF (README must not block on the GIF).

## Release pipeline: goreleaser on tag push
Status: accepted (M7 Publishing; brew tap + nix flake deferred — v1 installs are curl-binary + `go install` only)
Why: "Single static binary" is the distribution promise (DESIGN §3) but nothing today produces one; a tag-triggered pipeline is the difference between a repo and a product.
Sketch: `release.yml` on `v*` tags runs goreleaser: linux/darwin × amd64/arm64 with `CGO_ENABLED=0` pinned **and verified static** (`file`/ldd check — cgo-free today doesn't stop a future dep reintroducing it), archives + `SHA256SUMS`, GitHub-native artifact attestation, release creation gated on green CI, version/commit/date via ldflags into one `internal/version` package. A smoke step executes each built binary (`kira version` + a tiny testscript run — this, not vhs tapes, is release smoke). Brew tap and nix flake are deferred: each is a permanent maintenance surface (tap repo + PAT custody contradicts the sketch's own no-key-custody pitch; nix vendorHash churns per go.sum change) for zero users, and a stale formula is exactly the version-skew hazard the min_version guard exists to prevent.
Caveats: cross-compiled darwin artifacts ship unexecuted unless a macos smoke runner is added — state it in the release doc rather than implying coverage.

## `kira version` with contract handshake
Status: accepted (M7; `index_schema` dropped; `json_contract` sourced from the golden-suite constant)
Why: Two frontends (nvim health.lua handshake is already specced, 06) and users' bug reports all need one authoritative, machine-readable answer to "what am I running and what contract does it speak?".
Sketch: `kira version --json` → `{version, commit, date, json_contract}` from the ldflags-fed `internal/version` package; human form is one line. `index_schema` is dropped from the payload — 01-architecture makes index-schema mismatch a silent self-heal rebuild, so exposing it invites a false external dependency on rebuildable cache state. `json_contract` is what nvim and other tooling assert against — decoupled from the binary's semver, bumping only on a breaking shape change — and is sourced from the same constant the golden contract suite asserts against (a hand-bumped integer is a promise with no enforcement; the first forgotten bump makes the handshake lie). The `version --json` shape enters the frozen golden corpus in the same change; `version` must work with no `.kira/` present. Additive command, frozen-contract safe.
Caveats: `json_contract` is a floor check only — nvim will still capability-probe for additive features.

## Repo-format compatibility policy + `kira.min_version` guard
Status: split — codec fail-safe half implemented (M3.5 WP-3.5.3); policy doc + `min_version` guard remain (M7 WP-7.3); full byte-preservation deferred to its own design
Why: One repo, N teammates, N binary versions — the real upgrade path isn't "migrate my files", it's version skew: an old binary silently mangling frontmatter written by a newer one is the scariest failure a git-native tool can have.
Sketch: The original sketch's premise is false — the codec regenerates the entire frontmatter from struct fields (there is no in-place line rewrite in the package), so unknown keys are destroyed on every rewrite today. v1 policy is fail-safe refusal, not preservation: `Parse` records unknown frontmatter keys; unknown link types downgrade from ParseError to recorded-unknown (unblocking reads); every write path refuses with exit 3 + upgrade hint when unknowns are present — full destruction-prevention at a fraction of preservation's cost. True byte-for-byte preservation conflicts with the canonical-form invariant (fixed schema order, Parse/Serialize as property-tested inverses, one-line-diff guarantee) and needs an emit-position decision — deferred to its own design. Config gains optional `kira.min_version:`; an older binary exits 3 with "repo requires kira ≥ X (you have Y): <install hint>" before any write, with a light-scan fallback for the one key when a future config is otherwise unparseable. `doctor` warns on unrecognized keys/blocks; `kira migrate` stays the breaking-change escape hatch.
Caveats: judge-verified — `Parse` consumes only the fixed known key set and drops the rest; an unknown link type is a ParseError today, blocking even reads (parse.go:224); config loading is non-strict (config.go:311), so pre-guard binaries ignore `min_version` forever — guard + write-refusal must ship in the FIRST tagged binary.

## Demo assets as code: vhs tapes in the repo
Status: accepted (M7; `.tape` scripts committed only, rendered GIFs as release assets; blocked on the tour seeder — M4)
Why: A terminal tool is sold by its GIF; hand-recorded GIFs rot with every UI change and never get re-recorded — a stale screenshot is the universal smell of a dead project.
Sketch: charm's `vhs` .tape scripts under `docs/demo/`, each running against a repo produced by the tour seeder — commit the scripts only, never GIFs to main: regenerated binary blobs permanently bloat clone history, and vhs output is not byte-deterministic (font raster, timing, ttyd/ffmpeg versions), so every regen is a guaranteed-different multi-MB blob even with zero visual change. Rendered GIFs publish as release assets (or an orphan branch) referenced by stable URL; README embeds those URLs. v1 scopes to 1–2 tapes; regeneration is a **required release-checklist step** — a manual step nobody must run is a hand-recorded GIF with extra tooling. Tapes are docs assets, outside the frozen golden corpus per the batch-3 ruling. The "tapes double as release smoke" claim is cut: kira output embeds ULIDs/timestamps, so tape-output assertions are flaky by construction — release smoke is the pipeline's own testscript step.
Caveats: blocked on the tour seeder, accepted but unbuilt (targeted M4 per the tour ruling).

## README as the product page
Status: accepted (M7, last inside the milestone — after install paths are real)
Why: There is no README today; for a CLI tool the README is the landing page, the pitch, and the install doc — it decides adoption before the binary is ever downloaded.

# Git-native delight

All 6 refined 2026-07-12 by 4-assessor fan + judge; ranked value-for-cost: workon > blame > diff > time-travel (list/board) > inbox > standup — all six accepted, five modified (standup survives 3–1 with the minority's demotion folded in). Cross-cutting rulings: WP-2.3 exposes one internal event-query API where every derived event row carries a `source_kind` (`user|merge|auto-merge|doctor|closes-auto`) computed once at derivation — blame's flags, standup's `degraded_n`, inbox's filter, and stats' existing degradation all consume that one column, never per-verb squash-detection; one identity mapping — `people.known` gains emails (02 amendment), consumed by inbox `me`, standup `--author`, and blame display, landing with whichever ships first; one batched tree-ish loader (item set + `config.yaml` at any tree-ish via `ls-tree` + `cat-file --batch`, ULID-keyed, alias-aware), built once in the diff/`--at` WP; every new `--json` shape enters the frozen golden corpus in its shipping change **including** its `degraded`/`source_kind` markers — retrofitting honesty flags into a frozen contract is a breaking change; the `{ticket, branch}`-scoped active pointer + alias-aware pattern resolution amend 07 §3, fixing a latent mis-trailering bug in the existing spec independent of workon shipping.

## `kira blame <id>`: field-level provenance
Status: implemented (M2 — new WP-2.6, sequenced after WP-2.3's event shape settles; the launch-post flagship)
Why: "Who set this to P0, when, in which commit?" is the question every triage argument ends on; JIRA buries field history in a paginated audit tab — kira can answer for every field in one shot because git already recorded it.
Sketch: A GROUP BY over the event cache 08 §1 already mandates: keep the *latest* event per frontmatter field, render one line each — `priority: P0 · shivam · 2026-07-10 · a1b2c3d "triage pass"`. Attribution honesty from day one: a field whose latest event's source commit is a squash/merge, a `kira: auto-merged` commit (authored by whoever ran sync, 03 §7), or a doctor/Kira-Closes synthetic renders a `via merge` / `via auto-merge (ran by X)` flag, from the shared `source_kind` column; `--json` carries the per-field `source_kind`/`degraded` marker and enters the golden corpus in the shipping change. `Enter` on a line in the TUI detail pane jumps to `git show <sha>` — the audit trail is the actual commit, not a paraphrase.
Caveats: a confidently wrong name is worse than `log`'s same weakness — log shows a stream a reader can eyeball for gaps; blame's one line *is* the answer, so unmarked ship is wrong; the frozen corpus makes retrofitting the markers a contract break, hence day-one.

## Time travel: `--at <ref|date>` on list/board/show
Status: implemented (loader + `list`/`show --at` post-M2, same WP as `kira diff`; `board --at` lands with WP-4.2's `--plain`; `stats --at` cut from v1)
Why: "What did the board look like before the release?" / "show me the sprint as of Monday standup" — JIRA structurally cannot answer; a git-native tracker gets it for the price of `git show`.
Sketch: `--at` resolves a ref or date (`git rev-list -1 --before=<date> <landed_ref>` — the landed-ref anchor documented in `--help`) and loads the item set through the shared tree-ish loader — one batched `ls-tree` + `cat-file --batch` subprocess, never N× `git show` spawns — into the in-memory store: the linear-scan path that already serves index-less repos, no index involvement, read-only by construction (mutating verbs reject the flag). `config.yaml` loads from the same tree-ish — an old board rendered with today's workflow states breaks on renames; config time-travels with the tickets. User-supplied IDs resolve against the target tree's number/alias state, annotated when a display number maps to a different ULID than it does today (renumber between then and now). Board reuses `--plain` render. The killer composition: `kira board --at v0.9 && kira board` as a release retrospective in two commands.
Caveats: `stats --at` is a different feature smuggled in under the flag — 08 §1's only data source is the HEAD-derived event cache, and as-of-ref needs ancestry-truncation of that stream (no ancestry column exists; timestamp filtering is wrong under rebase); if ever wanted, respec it as an explicit slow-path full re-derivation at the ref with a degraded warning, never a cache filter.

## `kira diff <ref>`: semantic backlog diff
Status: implemented (post-M2 — new WP after WP-3.2, shared with the `--at` tree-ish loader; not release-blocking)
Why: A PR that moves three tickets and closes one shows as YAML hunks today; reviewers of ticket changes (DESIGN's own PR-review story, 07 §5) deserve the semantic summary — "what does this branch do to the backlog?".
Sketch: A two-snapshot parsed-struct compare, not a reuse of existing differs: enumerate + parse items at two tree-ishes (default: merge-base vs ref, stated explicitly) via the shared loader, pair by ULID (the filename), field-compare per pair through WP-3.2's scalar/list/body field-classification table (shared, never duplicated), created/deleted from unpaired ULIDs; render grouped one-line events. Noise suppression: a post-merge doctor renumber between the refs renders as one `renumbered KIRA-143 -> KIRA-151` line resolved via aliases — never as spurious number/aliases field-events on every touched item, or renumber churn drowns the real signal on exactly the multi-branch repos this targets; bare `updated`-timestamp churn suppressed likewise. Honest cap kept: body-text changes summarize as "body edited (+N/-M lines)", never a prose diff. `--json` for tooling; compose with anything (`kira diff origin/main` pre-push = "what am I about to publish"); immune to squash loss — it compares endpoint states, not the commit path.
Caveats: the original reuse claim is false — WP-2.3's differ consumes sequential `git log -p` patches (03 §5) and WP-3.2's engine is three-way per-file; neither is a two-tree comparator, so the tree-ish loader is new (small) code, built once and shared with `--at`.

## `kira workon <id>`: ticket-to-branch in one verb
Status: implemented (M3 — new WP after WP-3.1 hooks; pre-release path — the adoption lever, rank 1 of the six)
Why: The git-native loop today is four commands (branch, checkout, move --activate, remember the slug); collapsing it into one verb makes the ticket → branch → auto-trailer flow the path of least resistance — the moment the linking machinery starts paying rent invisibly.
Sketch: `kira workon KIRA-142 [--worktree <path>]` = idempotent per-*ticket* branch: match existing branches on the configured pattern prefix (`kira/kira-142-*`) and switch, else create with the slug helper — never per-slug, or a title edit mints a duplicate branch. Slug and the prepare-commit-msg branch pattern generate from one casing config key. Sets `.kira/.cache/active` as a `{ticket, branch}` pair; the hook honors it only when HEAD's branch matches the recorded branch, else falls through to branch-pattern inference. Hook pattern resolution is alias-aware — post-renumber branch names carry retired numbers (07 §4 guarantees resolution). Fires the configured doing-transition (skippable via `--no-move`; same validated core path). With `--worktree`: `git worktree add` instead of checkout — `.cache/` lives in each worktree's checkout, so every worktree carries its own active pointer for free; parallel tickets in parallel worktrees each auto-trailer correctly with zero new state.
Caveats: judge-verified 07 §3 conflicts the sketch had to resolve — the hook table shows an uppercase pattern (`kira/KIRA-142-*`) vs the lowercase slug (pin one casing source or the branch-name fallback never fires), and (a)-before-(b) source ordering plus a never-cleared pointer makes wrong trailers the default failure mode after a manual `git checkout` — the branch-scoped pointer amends 07 §3 in the same change and fixes that latent bug independent of workon; a post-checkout hook that clears the pointer was rejected as strictly more machinery for the same hole.

## `kira inbox`: on-sync digest of what concerns you
Status: accepted (M6 stretch — post-M3, needs sync + WP-2.3; design-blessed, DESIGN §3 names it)
Why: DESIGN §3 names an on-sync digest as the in-idiom notification substitute but never sketches it; this is the no-server answer to "what happened while I was gone" — pull-based, like git itself.
Sketch: The watermark is a **landed-ref commit SHA** in gitignored `.kira/.local/` — a per-clone *user state* carve-out, never `.cache/` (event-cache rows reshuffle on `index --full`/auto-rebuild, an advertised-safe operation and the WP-2.1 gate test; a SHA is stable) and never "index state". `kira inbox` reads the derived transition/comment events since that SHA from the event cache — not `updated` timestamps, which measure last-touch (the changelog rejection's exact critique) — filters to items where `owner=me` or I'm a comment-thread participant, groups by item, and advances the watermark on explicit ack or display-to-a-tty (`--peek` never marks). Identity prerequisite: `people.known` entries gain an emails list (or honor `.mailmap`) via a small 02 amendment — `owner=me` is undefined today; shared with standup. Watermark failing `merge-base --is-ancestor` ⇒ "history rewritten, showing last N days", never replay-all-as-unread; lost watermark ⇒ everything-unread; fresh clone initializes to HEAD (unread-nothing), never epoch. `sync` ends by printing "3 items concern you — kira inbox"; the `notify.exec` hook (DESIGN §3) gets the same digest as JSON on stdin, closing the team-channel story without a server.
Caveats: per-clone divergence (N machines, N inboxes) is accepted and documented as a decision — read state is per-clone, like the reflog; `.cache/active` already violates 03's purity claim, so the `.local/` carve-out lands with an explicit grandfather decision for it and reconciled 03 §1/§2 contract text.

## `kira standup`: author-grouped activity recap
Status: accepted (M6 stretch — sequenced strictly after inbox's identity mapping; contingent on dogfooding)
Why: The daily standup ritual is reconstructing yesterday from memory; kira's event stream plus linked commits already hold the true answer per person — a query JIRA needs a marketplace plugin for.
Sketch: `kira standup [--since 1d] [--author me|all]` reads the index's transition-event stream (close/move times from events, not last-touch) plus the commit_links table over the window, grouped strictly by **git commit author** — and the header says so, never implying kira `owner`: "shivam — moved KIRA-142 → REVIEW, closed KIRA-138, 4 linked commits (3 on KIRA-142)". `--author me` is blocked on the inbox identity mapping (`people.known` emails). Events whose source commit is a squash/merge/auto-merge/doctor commit count per-author as `degraded_n` via the shared `source_kind` column — never silently credited to the merger. Human render + `--json` (into the golden corpus with its degraded markers). Pure index read, no new data, no git walks at query time.
Caveats: survived 3–1 over a reject — 08 §5's non-goal rejects charting/dashboards, not kira-owned textual reports (`stats` is one), so it wasn't decisive; but a standup line naming the wrong person is the one format where the name *is* the payload — in squash-merge shops (DESIGN open question 8) the numbers are wrong by construction and the output must say so; contingency: if dogfooding shows it's just `kira log --json | group-by`, demote to a documented `--json` recipe instead of a verb.

## Dynamic shell completion from live data
Status: implemented (M3.5 WP-3.5.4)
Why: `kira move KIRA-<TAB>` completing to real ticket IDs with title hints is the single biggest daily-driver ergonomic after the TUI itself — and kira's `<10ms` cold start is the rare architecture where per-keystroke subprocess completion is actually viable (gh and kubectl pay 100ms+ for the same trick).
Sketch: cobra `ValidArgsFunction` on every ID/enum position: item IDs complete from the index with the title as the completion description (`KIRA-142\tFix race in snapshot merge`), `move` targets complete to states reachable from the item's *current* state (same adjacency map the TUI popup uses), `--label`/`--priority`/`--filter` complete from config vocab. **Serve stale, never silent-on-stale** (3–1 ruling): the cache is stale after *every* pull, so degrade-to-empty makes completion vanish exactly when teammates' new tickets arrive, indistinguishable from breakage — stale IDs are harmless suggestions the subsequent command validates; only an *absent* index degrades to no-suggestions. The `__complete` path opens the index **read-only with no staleness check or reindex** (a git subprocess eats the budget; no SQLite write lock, so a concurrent write and a TAB never race) — the one deliberate, explicitly-documented exception to 04-cli's implicit-refresh invariant. "Never a rebuild" extends to never the linear-scan fallback (parsing 1k files per TAB is where the budget dies). Prefix-filter server-side via cobra's `toComplete`, cap ~200 candidates, `ShellCompDirectiveNoFileComp`.
Caveats: bash renders description hints only under bash-completion v2 (zsh/fish are the real targets); `move`-target completion against a stale cache may offer a transition the command then rejects — intended and documented.

## Config-defined command aliases
Status: accepted (M6 stretch, CLI-only v1 — the flagship `mine` case is ~80% shipped via `filters:` + `list --filter mine`; aliases add argv sugar and shell-rc permanence, not capability; rank 4)
Why: `git config alias.*` muscle memory is the power-user contract for any git-adjacent tool; `kira mine` beating `kira list 'owner=me AND state!=DONE'` daily is how a scripting surface earns shell-rc permanence.
Sketch: **CLI-only v1**: config `aliases:` map (`mine: list --filter mine`, `wip: list state=DOING`) expanded once pre-cobra as an argv splice with git's rules by reference (built-ins never shadowed, recursion refused), git-style `!shell` **explicitly excluded** — argv splice only — and shell-style word splitting/quoting of alias values specified. Forward-compat shadowing warning on config load and in `kira alias list` when an alias name collides with a built-in — a future release adding a real command must not silently repurpose muscle memory. Aliases feed dynamic completion and the error contract's edit-distance suggester, both resolving *through* the expansion. Golden test pins alias transparency: `kira mine --json` emits exactly `list`'s frozen shape.
Caveats: the original "one definition, three frontends, zero extra code" headline is false (judge-verified) — the TUI `:` bar shares the argv *grammar* but dispatches item-scoped to internal/core, never the CLI pre-dispatch splice, and list-shaped aliases are a category error in an item-scoped bar; TUI support only as a follow-up that hoists expansion into the shared dispatch layer, restricted to aliases whose expansion is a valid item-scoped `:` command — until then the doc says CLI-only. Reconciles the `kira ls` pruning stance: kira ships one name per command, users may add their own — support answer is "check your aliases", not a bug.

## Grouped list output: `list --group-by owner|sprint`
Status: accepted (M6 stretch — team-shaped value, post-dogfood; v1 keys `owner|sprint` only; rank 5)
Why: "Who is carrying what" is the team question the flat list can't answer without N queries; grouped sections deliver the workload view leads actually want with zero new TUI surface — and no swimlane-board complexity.
Sketch: Presentational grouping over the existing query result: section header per group value (`— alice (4) —`), items rendered by the normal row renderer. **v1 keys: `owner|sprint`** (optionally `label`): `--group-by epic` errors with a hint to `--tree`, `--group-by state` errors with a hint to `board --json` — unrestricted keys would freeze a third/fourth grouping encoding into a contract that already defends "two shapes, deliberately" (04-cli). `--group-by` and `--tree` mutually exclusive — exit 1, never silent precedence. `--json` reuses the existing flat-bucket encoding — `groups: [{key, items:[ulid,...]}]`, ULID refs indexing the sibling `items[]` array, same shape family as the frozen `tree` key — so the contract gains a key, not a new grouping grammar; enters the golden corpus in the shipping change. Group ordering defined: owner lexicographic, sprint in config-declared order, `(none)` last; `ORDER BY` applies within groups only.
Caveats: recorded revisit trigger for a `state` key — board does not compose with arbitrary `--filter`/query expressions, so if dogfooding demands filtered state-grouping the key is added later, additively; the `ORDER BY owner` + `--group-by owner` no-op is documented, not surprising.

## Performance budgets enforced in CI
Status: implemented (non-gating report; CI gate lands at M7 WP-7.7)
Why: `<~10ms` cold start is a hard DESIGN §3 constraint and the nvim plugin's survival condition — yet nothing measures it; unenforced perf promises decay silently, and "verified every commit" is a publishable trust signal no tracker competitor makes.
Sketch: **Same-job A/B, relative not absolute**: build merge-base and HEAD in one CI job, run interleaved, gate on benchstat median-of-N (N≥10) *ratio* at 2× — runner speed cancels. Deterministic noise-free proxies alongside: git-subprocess spawn count per command on the 1k fixture, and a scaling-ratio check across 100/1k/5k fixtures — an accidental O(items×history) walk shows as a spawn-count diff or superlinear growth regardless of runner, exactly the regression class targeted. At most one absolute smoke ceiling as an order-of-magnitude tripwire (cold start < 50ms), never 2×; checked-in numbers survive only as a non-gating trend artifact. Lands as a non-blocking report first; promotes to failing only after weeks of stability. Fixtures reuse the tour-seeder machinery (M4) grown with a parametric size knob — the tour ruling specced it at ~6 tickets.
Caveats: the original design — a checked-in absolute wall-clock baseline with a 2× threshold on shared runners — is the canonical flaky perf gate (unanimous): runner-generation drift and CPU steal span 2–4× on a ~5ms spawn, and flake-driven baseline bumps ratchet the budget until the gate is ignored, worse than none.

## Crash containment: terminal restore + "your data is safe"
Status: implemented (M4 — ships with the first TUI WPs, folded into WP-4.1; crash exit code 4 reserved earlier, in the M3.5 error-contract amendment to the 04-cli §1 table, so wrappers are stable before the handler exists; rank 2)
Why: A panicking TUI that leaves the terminal in raw mode with no cursor is the single worst robustness impression a terminal tool can make — and kira has an honest, architecture-backed answer to "did that crash eat my tickets?" that deserves saying at the moment of failure.
Sketch: Terminal restore rides **bubbletea's default catchPanics path** — never disabled and reimplemented; v1.3.10's coverage is pinned empirically, with the testscript panic injected **inside a `tea.Cmd`** (Cmd-goroutine recovery is version-dependent), not just in Update or main. bubbletea's recovery/re-panic funnels into main's recover, which prints three fixed lines — what crashed (one-line panic), where the full trace went (`.kira/.local/crash-<ts>.log`, the per-clone user-state carve-out the inbox ruling established; `os.TempDir` fallback when `.kira/.local/` doesn't exist, since panics cluster before repo discovery — print where it actually went), and the data-safety statement (true by construction: atomic writes (03), plain files in git, worst case one uncommitted file, `kira doctor` verifies); main's recover stays as the belt for the non-TUI CLI path. **No bare `go func` in TUI code**: one panic-forwarding recover-wrapper helper mandated for every app-spawned goroutine (lint/convention) — a raw goroutine panic is structurally uncatchable, and the containment claim is scoped honestly to say so. The crash handler itself must be panic-proof. Exits 4.
Caveats: the original "top-level recover in main" sketch cannot work (unanimous) — Go `recover` is per-goroutine, and bubbletea runs Update/View in Program.Run's loop and every `tea.Cmd` on its own goroutine; as sketched, the flagship failure (panic in a Cmd/worker) bypasses containment while the injected-panic test passes and the real case still wedges the terminal.
