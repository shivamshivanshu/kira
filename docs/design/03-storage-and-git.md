# Storage & Git

**Scope:** the `.kira/` layout, canonical-vs-derived data split, file write invariants, history derivation, commit behavior, and merge strategy.
Part of the kira design set — see [DESIGN.md](../../DESIGN.md) for decisions and rationale.

## 1. `.kira/` directory tree

```
.kira/
  config.yaml                        # tracked: project key, workflows, labels, commit mode, trailers
  tickets/
    01J8X8Q7RZTN5Y3VXW2A9K4E7F.md    # one file per item; filename = immutable ULID
    01J8X7B1Q2W3E4R5T6Y7U8I9O0.md    # epics are `type: epic` in the same dir/schema
  templates/
    ticket.md                        # tracked: $EDITOR skeleton, user-customizable
    epic.md
  hooks/
    post-merge                       # tracked script: runs `kira doctor --fix` (ID reconciliation)
    prepare-commit-msg               # tracked script: auto-appends Kira-Ticket trailer (opt-in)
    pre-commit                       # tracked script: opt-in, installed by `hooks install --with-pre-commit` (proposed)
  .cache/                            # GITIGNORED — fully derived, delete-safe, never authoritative
    index.db                         # sqlite: items, labels, links, commit_links, fts5, event cache
    meta.json                        # {schema_version, last_indexed_head_sha, per-ref trailer watermarks, dirty-hash}
    active                           # local {ticket, branch} pointer for trailer auto-insert — grandfathered user state, see §2
    lock                             # advisory lock for concurrent kira invocations
  .local/                            # GITIGNORED — per-clone user state, NOT derived, never synced (e.g. inbox read watermark)
  .gitignore                         # generated by init: ".cache/" (gains ".local/" when the carve-out ships)
```

Per-entry notes:

- **`config.yaml`** — tracked, human-editable, the single source of workflow/label/commit-mode truth. See [02-data-model.md §9](02-data-model.md#9-example-config).
- **`tickets/`** — flat, one file per item, tickets and epics mixed (schema distinguishes via `type:`). No subdirectories, no per-item directories — see [02-data-model.md §2](02-data-model.md#2-unified-ticketepic-decision).
- **`templates/`** — tracked so a team's `$EDITOR` skeleton (default section headers, boilerplate acceptance-criteria checklist, etc.) is itself version-controlled and shareable, not a local dotfile.
- **`hooks/`** — tracked scripts, *not* live in `.git/hooks/` by default (git never auto-installs from a tracked path). `kira hooks install` copies/symlinks these into `.git/hooks/`, chaining onto any existing hook of the same name and refusing rather than clobbering. Full mechanics in [07-git-integration.md §3](07-git-integration.md#3-hooks).
- **`.cache/`** — gitignored, derived: deleting the whole directory is always safe. One grandfathered exception — `active` is per-clone user state, not a derived view, kept here because losing it is harmless (trailer auto-insert disengages until the next `--activate`/`workon`), so delete-safety still holds. New user state never lands in `.cache/`; it goes in `.local/`. See [§2](#2-canonical-vs-derived-split).
- **`.local/`** — gitignored per-clone **user state**, explicitly *not* derived: it records what this user in this clone has done (e.g. the inbox read watermark — a landed-ref commit SHA, stable where index state is not) and cannot be rebuilt from tracked files. Losing it is designed to be low-stakes (lost watermark ⇒ everything shows unread; a fresh clone initializes to HEAD). Never synced — read state is per-clone, like the reflog.
- **`.gitignore`** — generated once by `kira init`, contains `.cache/` (and `.local/` once the carve-out ships). Ordinary tracked file after that; users may extend it.

## 2. Canonical-vs-derived split

**Canonical**: the git-tracked markdown+YAML-frontmatter files under `tickets/` (plus `config.yaml`). This is the only truth. Every other representation — sqlite index, TUI state, nvim buffers — is a view over these files and can be regenerated from them.

**Derived**: `.cache/index.db` (sqlite). Gitignored, rebuildable from the tracked files plus git history at any time via `kira index --full`, and never a merge participant (it isn't tracked, so git never sees it).

**Local user state** — the third class, distinct from both: per-clone, gitignored, neither canonical nor derived. Home: `.kira/.local/` (e.g. the inbox read watermark). Unlike `.cache/`, it is not rebuildable — it records what *this user in this clone* has done — but every entry must be designed so losing it is low-stakes. `.cache/active` predates this split and is user state living in `.cache/`: it is **grandfathered in place** rather than migrated — losing it merely disables trailer auto-insert until the next `--activate`/`workon`, so the "deleting `.cache/` is always safe" invariant and its gate test (delete `.cache/` ⇒ identical results) hold unchanged.

Rejected alternatives:

| Alternative | Rejection reason |
|---|---|
| sqlite as canonical store | A binary blob defeats "git is our database" literally: no 3-way merge, no `rg`/grep, no reviewable PR diffs. Every edit becomes an opaque binary diff. |
| Plain files only, no index | Fine to ~1k items (linear scan), but query patterns (`kira list --query`, `kira stats`, label/state filters) degrade linearly past that — see [§8](#8-scale-expectations). The cache exists purely to make those queries fast without changing what's canonical. |

## 3. Writer format invariants

This is the keystone of the merge design ([§7](#7-merge-strategy)): a diff-minimizing file format so independent single-field edits don't collide.

- **Block-style YAML frontmatter**, never flow style (`{a: 1, b: 2}`).
- **One key per line.**
- **Fixed key order** — always the order in the [02-data-model.md §1](02-data-model.md#1-item-schema) field table. The writer never reorders keys, even when re-serializing the whole document.
- **In-place line rewrites** — changing `state` rewrites only the `state:` line; every other line in the file is byte-identical before and after.

**Invariant**: one field edit ⇒ one-line git diff. This is what makes different-field edits on the same ticket (e.g. branch A bumps `estimate`, branch B changes `labels`) merge cleanly as ordinary non-overlapping line changes, with no kira-specific merge logic required.

Enforced by byte-stability golden tests: write a fixture file, apply a single-field edit through the writer, assert the diff is exactly one line. Any writer change that reorders keys or reformats untouched lines fails CI immediately.

**Version-skew compatibility stance (v1).** The codec is canonical-form: serialization regenerates the full frontmatter from parsed struct fields, so keys an older binary doesn't know (written by a newer one) are dropped on rewrite today, not preserved. The v1 policy is fail-safe refusal, not preservation (M3.5): `Parse` records unknown frontmatter keys, unknown link types downgrade from parse errors to recorded-unknowns so reads still work, and every write path refuses with exit 3 + an upgrade hint while unknowns are present. Full byte-for-byte preservation of unknown keys conflicts with the fixed-order canonical-form invariant above (and the Parse/Serialize inverse property) and is deferred to its own design. Separately, `config.yaml` gains an optional `kira.min_version:`: an older binary exits 3 before any write when the repo demands a newer version, falling back to a light scan for that one key if the config no longer parses. Because config loading is non-strict, binaries released without the guard ignore the key forever — the guard and the write-refusal must ship in the first tagged binary.

## 4. Atomic writes

Every mutating command writes via **temp file + rename** (write to `tickets/.<ulid>.md.tmp`, `fsync`, `rename` over the target) — never a partial in-place write. An advisory lock at `.kira/.cache/lock` serializes concurrent `kira` invocations against the same repo to prevent interleaved writers; it is cache-adjacent (gitignored) and purely local-process coordination, not a git-level lock.

## 5. History

**No stored history.** `kira log <id>` is entirely derived at read time:

```
git log -p -- .kira/tickets/<ulid>.md
```
piped through a frontmatter-aware structural differ that turns each commit's patch into field-change events (`owner: alice -> bob`, `state: TODO -> IN_PROGRESS`, comment-block additions), interleaved chronologically with trailer-linked commits pulled from the index ([07-git-integration.md §1](07-git-integration.md#1-commit-linking-convention)). This event stream is also the *only* input to cycle-time/lead-time/throughput stats — there is no separate telemetry store.

No `--follow`: the filename is the immutable ULID ([02-data-model.md §7](02-data-model.md#7-id-scheme)) and never changes, so a plain `git log -p` on that path is already complete. `--follow` is not merely unnecessary here — it is *unsafe*: its rename/copy detection pairs the file's add-boundary against structurally-similar sibling ticket files, so a ticket can inherit another's history (a content/SHA-sensitive, run-to-run-nondeterministic bleed). We omit it deliberately.

**Rejected: the founder's original snapshot-linked-list idea** — on every edit, snapshot the old item and link it to the new one in a simple in-repo linked list, giving an explicit version chain per item. Evaluated and rejected for four concrete reasons:

1. It duplicates git's own object model — git already stores every prior blob for this file, reachable via the commit graph; a parallel snapshot chain is redundant machinery solving an already-solved problem.
2. It bloats every ticket file (or spawns snapshot files) with data that provides no new information over `git log -p`.
3. It doubles the merge surface: now both the live file *and* the snapshot chain can conflict, for the same underlying change.
4. It drifts the moment anyone hand-edits a ticket file outside kira (fixing a typo, a manual `git revert`) — the snapshot chain silently stops matching reality, while `git log -p` never can, because it *is* reality.

The derived-view answer costs zero storage and inherits git's correctness, authorship, and timestamps for free — it is strictly better on every axis the linked-list idea was trying to address.

## 6. Commit modes

`config.yaml`'s `commit.mode` ∈ `auto` (default) \| `manual` \| `prompt`.

| Mode | Behavior |
|---|---|
| `auto` | Every mutating command (`create`, `edit`, `move`, `assign`, `link`, `comment`, `doctor --fix`, ...) writes its file(s), `git add`s exactly those path(s), and commits immediately with a structured message + trailer. |
| `manual` | Command writes and stages only. Read commands (`list`, `show`, ...) print an "N kira changes staged" nag if the working tree has staged-but-uncommitted kira paths. `kira commit [-m "msg"]` commits kira-touched paths only, on demand. |
| `prompt` | Same as `auto`, but asks (y/n) before each commit. |

Structured commit message format:
```
kira: KIRA-142 state TODO -> IN_PROGRESS
```
with a `Kira-Ticket: KIRA-142` trailer appended (trailer key is configurable — [07-git-integration.md §1](07-git-integration.md#1-commit-linking-convention)). The `kira:` prefix makes these commits trivially filterable (`git log --grep '^kira:'`) or squashable.

Guarantees, all modes: stages only the paths its own command touched (never `git add -A`, never unrelated staged files); never passes `--no-verify`; never pushes. Publishing is a separate, explicit act — see `kira sync --push` in [07-git-integration.md §5](07-git-integration.md#5-remote--collaboration-model).

**Open question for the founder**: whether `auto` should be the default at all versus `manual` (state changes ride in the developer's own fix commit instead of a separate kira commit). Both are fully supported; only the *default* is unresolved — flagged in [DESIGN.md](../../DESIGN.md).

## 7. Merge strategy

Layered avoidance, not clever merging — each layer eliminates one class of conflict by construction, making auto-resolution ([below](#auto-policy-field-level-three-way-merge)) rare and, when it does trigger, small (one or two fields, never a whole-file rewrite):

| Layer | Eliminates |
|---|---|
| File-per-ticket | Cross-ticket edits never touch the same file |
| Single-sided edges (parent-on-child, blocked_by-on-dependent) | Relationship edits never touch a second file — [02-data-model.md §3](02-data-model.md#3-edges) |
| One-key-per-line, fixed order | Different-field edits on the same ticket merge line-wise — [§3](#3-writer-format-invariants) |
| Append-only comment blocks | Comments only ever append, never edit in place, so the kira driver/`resolve` unions them by `id` with no content merge. Unlike the layers above this is *not* conflict-free by construction: concurrent end-of-file appends still collide under plain `git`, and it is the driver's comment-union that makes them clean — [02-data-model.md §4](02-data-model.md#4-comments) |
| Gitignored `.cache/` | Index is never staged, never a merge participant |

**Policy knob**: `merge.policy: auto` (default) \| `manual`, in `config.yaml`. The founder's requirement: users must never be dropped into raw git conflict markers on kira files, JIRA-simple by default. `manual` is fully shipped for teams that want a human decision recorded on every same-field collision instead.

### `auto` policy: field-level three-way merge

A same-field conflict (both branches touched, say, `state`) is resolved automatically by a shared policy engine, applied per field against base/ours/theirs:

| Field class | Fields | Rule |
|---|---|---|
| Scalar | `state`, `owner`, `reporter`, `priority`, `title`, `estimate`, `epic` | one side changed vs base → take it; both sides changed → later `updated` timestamp wins; exact tie → the **incoming/remote** side wins — deterministic either way; see the tie-break note below for what "incoming/remote" means precisely |
| List (set) | `labels`, `blocked_by` | three-way set merge: `(base − removals from either side) ∪ additions from both sides` |
| `aliases` | — | union of both sides only — removals are **never honored**, regardless of what base says, since a retired `number` must keep resolving forever ([02-data-model.md §7](02-data-model.md#7-id-scheme)); this is not the general list rule above |
| Comments | body `## Comments` blocks | union by comment `id`, sorted by `ts` — cheap for the engine since comments never edit in place, but a real merge step: concurrent appends collide under plain `git`, so the union is the driver's doing ([02-data-model.md §4](02-data-model.md#4-comments)) |
| Body prose | `## Description`, `## Acceptance criteria` | one side changed → take it; both changed → `git merge-file` line-level text merge; a genuine textual conflict inside that merge falls back to whole-body LWW by `updated` |
| `created`, `type` | — | immutable; both sides must already agree — schema invariant, not a merge decision |
| `updated` | — | `max(ours.updated, theirs.updated)` |
| `number` | — | never resolved at merge time by this policy at all; post-merge `number` collisions are a separate mechanism, repaired by `doctor --fix` — see [07-git-integration.md §4](07-git-integration.md#4-id-collision-repair-flow) |

**Tie-break role definition**: git's "ours"/"theirs" labeling flips depending on the operation — during `git merge`, "ours" is local and "theirs" is the side being merged in; during `git rebase` (which is what `kira sync` runs — [07-git-integration.md §5](07-git-integration.md#5-remote--collaboration-model)), the labels invert: the commit being replayed (the local work) is "theirs" and upstream becomes "ours". A literal "theirs" tie-break would therefore pick the *local* side under `kira sync` and the *remote* side under a plain `git merge`/merge-driver invocation — silently inconsistent. To keep the rule deterministic and identical on both paths, kira re-maps git's stage labels (index stages 2/3 in a merge driver's `%A`/`%B`, or the equivalent during a rebase) to absolute **local**/**remote** roles by checking whether a rebase is in progress (`.git/rebase-merge` or `.git/rebase-apply` present) before applying the tie-break — the exact tie always resolves to the **remote** side, however git itself is labeling ours/theirs at that moment.

**Why aggressive last-write-wins is safe here** — this is the keystone of the `auto` policy: git preserves both merge parents regardless of which value the policy picks. The losing write is never destroyed — it stays in the commit graph, visible via `kira log <id>` ([§5](#5-history)). Every auto-resolution is reported: in the `kira sync` summary, and as a structured commit, e.g. `kira: auto-merged KIRA-142 (state: REVIEW->DONE, took remote)`. Net effect: JIRA semantics (last write wins, no conflict UX to deal with) plus a git audit trail JIRA itself doesn't give you.

Delivery mechanism (how `auto` reaches both `kira sync` and plain `git merge`/`pull`, via a registered merge driver): [07-git-integration.md §5](07-git-integration.md#5-remote--collaboration-model).

### Deletion

v1 declares ticket deletion out of scope: there is no `kira delete`. A ticket file can still be hand-removed with `git rm`, though, and git only invokes a merge driver when **both** sides modified the blob — a `git rm` on one branch against a concurrent edit on the other is a modify/delete conflict, which git surfaces as raw conflict state regardless of `merge.policy`, because the driver is never called. This is the one case `auto` cannot intercept, and it's documented rather than silently gapped: `kira doctor` flags any dangling reference to a deleted-but-referenced ticket, and `kira resolve <id>` handles a surfaced modify/delete conflict after the fact — the edit wins by default (the deletion is reported, not silently applied), consistent with `auto`'s general bias toward never losing a write outright.

### `manual` policy (the original design, still shipped)

Same-field conflicts surface as a real, ordinary git merge conflict — not auto-arbitrated, on the reasoning that a state disagreement can be a decision a human should make. Resolved with `kira resolve <id> --interactive`: a field-aware picker that parses base/ours/theirs for the conflicting item and writes back one clean, invariant-respecting file. Not the default, since requiring a manual pick on every same-field collision is exactly the git-like friction the founder asked to avoid — but available for teams that prefer it.

`kira resolve <id>` (no flags), regardless of policy: applies the `auto` engine to any currently-conflicted kira paths — the escape hatch when a plain `git merge`/`pull` outside `kira sync` left conflict markers (e.g. no merge driver installed). Doubles as the audit surface for reviewing what an auto-merge did, pointing at `kira log <id>` for the full before/after.

ID collisions (two branches allocate the same `number`) are a separate mechanism — not merge-time, post-merge hook driven — see [07-git-integration.md §4](07-git-integration.md#4-id-collision-repair-flow).

## 8. Scale expectations

- File-per-ticket with no index: fine to roughly **~1k items** — `list`/`query`/`find` degrade to a linear scan over that many small files, which is still sub-second.
- Beyond that, the index (`.cache/index.db`) is what keeps `list`/`query`/`stats` fast; it's rebuilt incrementally per invocation ([07-git-integration.md §2](07-git-integration.md#2-incremental-scan)).
- These numbers are rough guidance for when to care about index health, not hard limits — nothing breaks past 1k items, queries just get slower without the cache.
