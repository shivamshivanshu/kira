internal/core/sync.go:195-213: when stash pop conflicts, git keeps the stash entry; the auto path calls s.autoResolve (stages via Resolve) then StepDone 'auto-resolved conflicts' with no drop — gitx has no StashDrop at all (grep: only hint strings mention it, three of them at sync.go:200,201,210). Manual/failed paths tell the user to 'git stash drop'; the successful auto path silently accumulates orphan stash entries and leaves resolutions staged, unlike a clean pop. Note git's merge machinery during a conflicted pop also leaves cleanly-merged paths staged, not just the kira paths Resolve stages — unstage via restore --staged accordingly if matching clean-pop semantics.

Also a coverage hole: sync_test.go covers only prepareTree and popStash basics; nothing drives pullRebase (sync.go:137-192: RebaseContinue error handling, terminal-unmerged abort, 100-iteration cap); no sync-with-remote fixture exists; workon.go untested.

Fix: add gitx.Repo.StashDrop() and call it after autoResolve leaves zero unmerged paths; optionally restore --staged the resolved paths.

Verify: integration fixture with a local bare origin + second clone producing divergent kira edits — clean rebase, auto-resolved modify/modify, manual-policy abort, non-kira-conflict abort; unit tests for doingTarget and workon branch selection.
