SplitComments (internal/codec/comments.go:100-107) cuts prose at commentBoundary and keeps only ParseComments output; ParseComments (:27-34) drops an unterminated comment and everything after; merge/body.go:11-18 round-trips all three sides — so an indented close marker or epilogue prose vanishes with no conflict. LintComments (:51-66) matches TrimSpace'd lines while ParseComments (:22,:28) requires raw exact matches: a lint-clean body can be invisible to the parser (doctor cannot detect the whitespace-indented-marker class or epilogue/interleaved prose; it DOES flag unterminated blocks and malformed markers). formatComment (:117-121) concatenates c.Author verbatim but commentOpen (:14) requires author=(\S+); cfg.People.Canonical returns configured names verbatim (datamodel/vocab.go:39-51), so "John Smith" produces an unparseable marker that flattens to prose on merge. AppendComment injects a spurious leading newline when the body starts with a comment (:96-98), breaking JoinComments∘SplitComments identity.

Fix (ordered — (a) depends on (d)):
(d) skip the separator when prose is empty;
(a) make SplitComments conservative: if JoinComments(SplitComments(body)) != body, return whole body as prose with comments=nil so merge falls to plain text;
(b) unify matching discipline: lint on raw lines, flag trimmed-only matches;
(c) validate/normalize ID/Author/Ts at the write layer (apply identity()'s '-' join to the Canonical branch; reject Body lines matching the marker grammar).

Verify: table tests — missing close, indented close, prose between/after blocks, marker-at-start byte equality; a direct LintComments table (currently zero direct tests).
